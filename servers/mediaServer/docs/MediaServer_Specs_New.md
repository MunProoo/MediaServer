# MediaServer 스펙 문서

작성일: 2026-03-25

## 1. 카메라 수별 권장 스펙 (핵심 결론)

아래 표는 **“100대 상시 스트림 유지 + WebRTC 그리드 동시 재생(시청자 1~2명) + 녹화 동시 6대”**를 기반으로 보수적으로 잡은 **권장치 결론**입니다.

### 1-1. 공통 권장

- Network
  - 최소 1Gbps, 권장 2.5Gbps
- Disk I/O 안정성
  - HDD 단독도 가능하나, 쓰기 지연/버벅임이 생기면 저장 구간부터 품질이 떨어질 수 있음
- RAM 여유
  - Go의 채널 버퍼/패킷 큐가 누적될 수 있으므로 “여유 있는 RAM”이 안정성에 직접 영향

### 1-2. 카메라 수별 권장(예시)

| 카메라(스트림) 수 | CPU 권장        | RAM 권장 | 네트워크 권장  | HDD 권장(30일 기준)           |
| ---------- | ------------- | ------ | -------- | ------------------------ |
| 10 ~ 24    | 6코어/12스레드 이상  | 16GB   | 1Gbps    | 4TB (6대 녹화 기준 여유)        |
| 25 ~ 49    | 8코어/16스레드 이상  | 32GB   | 2.5Gbps  | 6TB (여유/증가분 반영)          |
| 50 ~ 79    | 12코어/24스레드 이상 | 48GB   | 2.5Gbps  | 8TB                      |
| 80 ~ 100   | 16코어/32스레드 이상 | 64GB   | 2.5Gbps+ | 8TB ~ 12TB (10대까지도 노릴 때) |

설명:

- CPU는 “비디오 해상도 변환”이 아니라, 내부 복제/패킷 처리/Go 런타임/암호화/큐 처리 등에 의해 소모됩니다.
- 따라서 **동시 WebRTC 송출 + 동시 녹화 프로세스(6대)** 조합에 의해 결정되는 경향이 큽니다.

## 2. 용어/가정

- 카메라 등록 수: **100대**, 채널 수: **1**
- 카메라 스트림 품질(대략): **비트레이트 2 Mbps**, 해상도 1920x1080, 30fps
- WebRTC 시청자(동시 접속자): 최대 **1~2명** (관제 시스템 그리드에서 동시에 100 스트림을 재생)
- 녹화 동시 실행: **최대 6대 유지**(여유가 있으면 10대까지 운영 가능)
- 녹화 스토리지 실측:  
  - “**1대의 24시간 녹화 용량이 약 15GB**”

참고:

- WebRTC는 “시청자가 실제로 모두 100 스트림을 계속 재생”할 때 네트워크/메모리 부하가 최악이 됩니다.
- 녹화는 비디오 `copy` 기반이므로 “해상도 변환/트랜스코딩 CPU 폭탄”보다는, 프로세스/패킷 처리 + 오디오 인코딩 + HLS 파일 생성/암호화가 주된 비용입니다.

## 3. 미디어 서버의 주요 역할 (코드/동작 기준)

이 미디어서버는 RTSP 입력에 대해 아래 3가지를 수행합니다.

1. **RTSP 복제(팬아웃)**  
각 스트림은 카메라 `opt.URL`(원본 RTSP)에 한 번 접속하여 패킷을 수신하고, 내부적으로 여러 소비자(WebRTC/RTSP 등)에게 **복제 스트림** 형태로 전달합니다.
2. **RTSP 녹화(HLS, ffmpeg 기반)**  
녹화는 ffmpeg가 카메라 원본 RTSP에 직접 붙는 것이 아니라, 서버 내부 RTSP(`rtsp://localhost:<RTSPPort>/<streamID>/<channelID>`)를 ffmpeg 입력으로 사용해 HLS를 생성합니다.  
현재 설정은 비디오에 대해 `-c:v copy`(재인코딩 없음)이며, 오디오는 `-c:a aac`로 인코딩합니다.
3. **RTSP -> WebRTC 전송**  
WebRTC는 사용자 요청마다 생성되는 WebRTC muxer에 대해, 위의 **복제 스트림(내부 패킷 fan-out)**을 소비하여 전송합니다.  
즉 WebRTC는 “카메라 원본 RTSP”를 별도로 재접속하는 구조가 아니라, 서버 내부에서 복제된 스트림을 사용합니다.

## 4. 네트워크 권장 산정

### 4-1. 카메라 → 미디어서버 인입

- 100대 × 2 Mbps = **200 Mbps**

### 4-2. 미디어서버 → 브라우저 WebRTC 송출 (최악)

“100 스트림을 동시에 재생”하는 경우를 가정합니다.

- 시청자 1명: 100 × 2 Mbps = **200 Mbps**
- 시청자 2명: 100 × 2 Mbps = **400 Mbps**

### 4-3. 합계(오버헤드 포함 여유 권장)

- 시청자 1명일 때: 200 + 200 = **400 Mbps**
- 시청자 2명일 때: 200 + 400 = **600 Mbps**
- 여유(오버헤드/재전송/다른 트래픽): 약 20~30% 가정

따라서 권장:

- **최소:** 1Gbps
- **권장:** 2.5Gbps (안전 마진 확보)

## 5. 녹화 스토리지 계산 (실측: 1대/24h ≈ 15GB)

### 5-1. 단위

- 1대/일: **15GB**
- 6대 동시 녹화/일: 15GB × 6 = **90GB/일**
- 10대 동시 녹화/일: 15GB × 10 = **150GB/일**

### 5-2. 보관일수 예시 (기본 설정: retention_days=30)

#### (1) 동시 녹화 6대, 30일

- 90GB/일 × 30 = **2,700GB ≈ 2.7TB**
- HLS 분할/헤더/파일시스템 여유(권장 1.2배):  
  - 2.7TB × 1.2 = **약 3.24TB**

→ HDD 권장: **4TB 이상** (여유 포함하여 6TB 권장)

#### (2) 동시 녹화 10대, 30일

- 150GB/일 × 30 = **4,500GB ≈ 4.5TB**
- 여유 1.2배:
  - 4.5TB × 1.2 = **약 5.4TB**

→ HDD 권장: **8TB 이상**

### 5-3. 추가 고려사항(현장 오차)

- 카메라 설정(비트레이트 변동), 장면 복잡도에 따라 15GB/일은 증감 가능
- 암호화(HLS AES-128) 및 오디오 인코딩으로 인해 “실측 기반”이 가장 정확
- 따라서 운영 첫 1~3일 동안 실제 용량을 확인해 보정 계수를 적용하는 것을 권장

## 6. 운영 권장 체크리스트

1. 녹화는 “동시 6대”를 유지하고, 실제 누적 용량이 예상(15GB/24h)과 맞는지 1~3일 관측 후 보정
2. WebRTC 송출이 증가하는 시간대(피크)에 NIC/디스크/메모리 사용량 모니터링
3. HDD는 “용량”뿐 아니라 “연속 쓰기 성능”이 중요 (RAID/저속 장치 회피 등)
